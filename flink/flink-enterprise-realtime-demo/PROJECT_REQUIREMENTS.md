# 企业级实时数仓 Demo 项目说明

## 1. 项目目标

基于现有远程环境 `bigdata`，完成一个可落地、可运行、可验证的企业级实时数仓 Demo。

项目需要覆盖以下完整链路：

- 数据源来自 `MySQL` 和 `Kafka`
- 使用 `Flink CDC` 将 MySQL 数据实时同步到 `ODS` 层
- 使用 `Flink` 完成实时宽表构建
- 使用 `Flink` 完成实时指标计算
- 使用 `Flink` 完成实时告警任务
- 最终通过 `Doris` 查询所有产出的表
- 代码完成后，需要上传到远程服务器并完成联调测试
- 所有链路验证通过后，再输出架构图和流程文档

## 2. 目标环境

默认基于以下远程环境开展：

- 服务器连接方式：`ssh bigdata`
- 主目录：`/opt/module`
- 已部署组件：
  - Doris
  - Kafka
  - Hadoop 3.1.3
  - Spark
  - Flink 1.17.1

补充约定：

- 默认优先复用现有组件和目录
- 默认先检查现有环境，再进行新增部署或修改
- 默认避免高风险操作，例如无明确授权的大规模重启、删除历史数据、覆盖核心配置

## 4. 业务场景设计

本项目由我自行设计一套贴近企业场景的业务模型，并保证能完整体现以下能力：

- 主数据通过 MySQL CDC 持续进入实时链路
- 行为数据通过 Kafka 持续进入实时链路
- ODS 层承接原始明细
- DWD 层生成实时宽表
- DWS 层生成实时指标
- ADS / 告警层生成实时告警结果
- Doris 统一对外提供查询能力

建议采用“电商交易分析”作为示例场景，便于同时覆盖维表、事实流、宽表、指标和告警。

建议的数据设计方向如下：

- MySQL 维度表
  - 用户表 `user_info`
  - 商品表 `product_info`
  - 商户表 `shop_info`

- Kafka 行为流 / 事实流
  - 订单流 `order_events`
  - 支付流 `payment_events`
  - 浏览流 `page_view_events`

- ODS 层
  - `ods_user_info`
  - `ods_product_info`
  - `ods_shop_info`
  - `ods_order_events`
  - `ods_payment_events`
  - `ods_page_view_events`

- DWD 层
  - `dwd_order_wide`
  - `dwd_payment_wide`

- DWS 层
  - 实时 GMV 指标
  - 实时下单人数 / 支付人数
  - 商品维度销售排行
  - 商户维度交易指标

- 告警层
  - 支付失败率告警
  - 某商品 / 商户短时异常波动告警
  - 流量高峰告警

## 5. 技术链路要求

### 5.1 ODS 层

- MySQL 数据通过 Flink CDC 采集
- Kafka 数据通过 Flink Source 采集
- ODS 层需要保留相对原始的结构
- ODS 层数据需要能被后续 Flink 作业和 Doris 查询使用

### 5.2 实时宽表

- 至少实现一张核心宽表
- 宽表需要体现事实流与维度表关联
- 宽表需要能支撑后续指标和告警计算

### 5.3 实时指标

- 至少实现 2 到 4 个有业务意义的实时指标
- 指标需具备窗口、分组、聚合等实时计算能力
- 指标结果可落入 Doris 供最终查询

### 5.4 实时告警

- 至少实现 2 类告警规则
- 告警结果写入可查询表
- 告警不要求一定推送到短信、邮件、企业微信等外部系统，但必须有明确的告警结果产出

### 5.5 Doris 查询

- 所有核心表最终都应支持通过 Doris 查询
- 至少覆盖：
  - ODS 表查询
  - 宽表查询
  - 指标表查询
  - 告警表查询

## 6. 实施要求

### 6.1 代码实现

- 项目代码需要结构清晰，便于后续扩展
- 优先采用易于在现有环境中落地的实现方式
- 可以使用 Flink SQL、DataStream API，或二者结合
- 需要兼顾“能跑通”和“具备企业级演示价值”

### 6.2 部署验证

- 代码完成后，由我负责上传到 `bigdata` 服务器并运行验证
- 需要给出明确的执行步骤
- 需要完成端到端联调
- 需要确认最终表中确实产生了可查询结果

### 6.3 验收标准

以“链路真正跑通”为准，至少满足以下条件：

1. MySQL CDC 数据能够进入 ODS
2. Kafka 事件数据能够进入 ODS
3. 宽表能够成功生成
4. 指标表能够持续产出结果
5. 告警表能够产出告警记录
6. Doris 能够查询到上述各层表数据
7. 提供完整架构图和流程文档
